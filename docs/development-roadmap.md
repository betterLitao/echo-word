# EchoWord 开发周期规划

## 1. 拆分原则

- 以 `P0` 为主线，先完成最小可交付闭环，再进入体验增强与扩展能力。
- 每个周期结束都应产出一个“可运行、可演示、可验收”的版本，而不是只完成局部模块。
- 先打底座，再做翻译主链路，最后补齐系统集成、运营能力和扩展功能，减少架构返工。
- 双窗口、IPC、数据库、设置、权限、系统托盘等基础设施只在前期打一次底，后续按功能增量扩展。

## 2. 周期总览

| 周期 | 名称 | 主要目标 | 对应优先级 | 关键交付 |
|------|------|----------|------------|----------|
| Cycle 01 | 基础工程与应用骨架 | 建立 Tauri 双窗口、数据库、设置、路由、托盘骨架 | P0 前置 | 可启动的桌面应用骨架 |
| Cycle 02 | 单词翻译闭环 | 打通离线词典、音标谐音、单词弹窗、收藏闭环 | P0 | 第一个核心价值版本 |
| Cycle 03 | 句子翻译与触发链路 | 完成在线句子翻译、缓存/降级、快捷键、输入翻译 | P0 | 主翻译链路可用 |
| Cycle 04 | P0 收口与发布候选 | 完成划词自动弹窗、权限引导、设置闭环、稳定性加固 | P0 | MVP Release Candidate |
| Cycle 05 | P1 体验增强 | 补齐朗读、复制、历史、收藏列表、主题、快捷键自定义等 | P1 | 日常可持续使用版本 |
| Cycle 06 | 扩展集成与 P2 储备 | 完成词典管理、同步/目录能力、OpenAI/多引擎、HTTP API、更新能力，并规划 P2 | P1/P2 | 可扩展版本与下一阶段输入 |

## 3. 依赖关系

```text
Cycle 01 -> Cycle 02 -> Cycle 03 -> Cycle 04 -> Cycle 05 -> Cycle 06
```

- `Cycle 01` 是所有实现工作的地基，没有它不建议直接进入翻译能力开发。
- `Cycle 02` 为 `Cycle 03` 提供统一翻译结果结构、弹窗 UI 骨架和收藏数据模型。
- `Cycle 03` 完成翻译主链路后，`Cycle 04` 才适合处理权限、划词自动触发和 P0 验收。
- `Cycle 05` 与 `Cycle 06` 建议在 `Cycle 04` 通过回归后再推进，避免体验增强建立在不稳定主链路之上。

## 4. 功能映射

| 功能 | 周期 |
|------|------|
| Tauri 双窗口、路由、托盘、设置、Migration | Cycle 01 |
| 离线单词翻译、音标、谐音、一键收藏 | Cycle 02 |
| 在线句子翻译、缓存、防抖、降级链、全局快捷键、输入翻译 | Cycle 03 |
| 鼠标选中自动弹窗、首次引导、权限申请、P0 回归 | Cycle 04 |
| 发音朗读、复制、历史、收藏列表、主题、快捷键自定义、剪贴板监听、命名拆分 | Cycle 05 |
| 数据目录切换、同步冲突处理、词典下载、OpenAI、多引擎、HTTP API、自动更新、代理配置 | Cycle 06 |

## 6. 使用建议

- 若当前团队规模较小，可将每个周期控制在一次完整迭代内，优先保证“验收标准”达成，而不是并行铺开过多模块。
- 若团队规模较大，可在同一周期内部按“Rust 后端 / 前端 UI / 系统集成 / 测试验收”四条泳道并行推进。
- 若 `Cycle 05` 或 `Cycle 06` 任务量超出预期，可继续拆成 `05A/05B` 或 `06A/06B`，但建议保持当前顺序不变。
