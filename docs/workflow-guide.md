# 从零到一：用 Claude Code 驱动新产品的完整流程

> 本文以 **EchoWord**（轻量级划词翻译工具）为例，记录如何通过与 AI 对话，从一个模糊想法推进到可运行的桌面应用。覆盖需求讨论、竞品分析、技术设计、开发规划、编码实现、评审优化到功能测试的全过程。适合想了解"AI 辅助产品从 0 到 1"工作流的开发者。

---

## 一、整体流程概览

```
需求萌芽 → 市场调研 → 需求讨论 → 需求定稿 → 竞品对比 → 技术设计 → 开发规划 → 项目初始化 → 按周期开发 → 功能文档生成 → 评审优化循环 → 功能测试循环
```

分为 12 个阶段：

| 阶段 | 会话 | 做什么 | 产出物 |
|------|------|--------|--------|
| **第一阶段** | 会话 1 | 和 Claude 来回对话 5 轮，逐步明确需求 | `PRD.md` 初稿 |
| **第二阶段** | 会话 2 | 让 Claude 搜索竞品对比，找出 18 项不足 | `PRD.md` 定稿 |
| **第三阶段** | 会话 3 | 让 Claude 输出技术实现文档 | `technical-design.md` |
| **第四阶段** | 会话 3 | 让 Claude 拆分开发周期，输出各周期任务清单 | `development-roadmap.md` + `cycles/` |
| **第五阶段** | 会话 1 | 回到第一个会话，创建 GitHub 仓库，文档入库 | GitHub 仓库 + 完整文档 |
| **第六阶段** | — | 进入项目目录，启动 Claude Code | 新的工作环境 |
| **第七阶段** | 会话 4+ | 让 Claude 读取文档，按周期开发 | 可运行的代码 |
| **第八阶段** | 持续 | 每个周期一句"继续"推进 | 完整应用 |
| **第九阶段** | 会话 N | Codex 根据代码生成功能实现文档 | `feature-implementation.md` |
| **第十阶段** | 会话 N | Claude 评审功能文档 vs 原始文档，输出优化清单 | 优化清单 |
| **第十一阶段** | 循环 | Codex 按清单优化 → 更新文档 → Claude 再评审，直到没有优化点 | 稳定版代码 |
| **第十二阶段** | 循环 | Claude 给测试清单 → Codex 排查 → Claude 再给清单，直到测试通过 | 可发布版本 |

> **为什么用多个会话？** 每个阶段的任务性质不同，新会话能让 Claude 专注当前任务，避免上下文过长导致"记不住"前面的讨论。关键上下文通过项目文件（PRD.md、CLAUDE.md）在会话间传递。

> **为什么引入多模型协作？** 从第九阶段开始，工作流引入了 Claude + Codex 双模型协作模式：Claude 负责规划、评审和任务编排，Codex 负责代码实施。这种分工让"规划者"和"执行者"各司其职，减少单模型上下文过长导致的质量下降。

## 二、第一阶段：需求讨论（从模糊到清晰）

### 2.1 起点：一个朴素的需求

最初只是一个简单的问题：

> "有没有这样的软件——划词翻译，能显示音标，还能展示音标对应的中文谐音？因为我看不懂音标。"

这个阶段的关键是**不急着动手**，先和 Claude 充分讨论。

### 2.2 对话策略：逐轮收敛

需求讨论不是一次性完成的，而是通过多轮对话逐步收敛：

**第 1 轮：市场调研**
- 让 Claude 搜索现有翻译软件
- 结论：市面上没有"音标中文谐音"功能的翻译工具
- 决策：**这是一个值得做的差异化功能**

**第 2 轮：技术可行性**
- 问 Claude："如果我想做一个这样的小插件，可以吗？"
- Claude 给出三种技术方案（Electron / Tauri / 浏览器插件）及对比
- 决策：**选择 Tauri（轻量化）**

**第 3 轮：核心功能讨论**
- 数据同步方案（坚果云 WebDAV vs 本地文件同步）
- 收藏功能的数据结构
- 全局划词的触发方式
- 产品命名
- 决策：**坚果云本地同步、全局快捷键+鼠标选中双触发、命名 EchoWord**

**第 4 轮：细节确认**
- 翻译 API 选择（离线词典 + 在线 API 混合方案）
- 音标谐音的精度策略（混合标注方案）
- 弹窗交互形式（跟随鼠标气泡）
- 补充建议（发音朗读、历史记录、快捷键自定义、输入翻译）

**第 5 轮：最终确认**
- 逐项确认所有决策
- Claude 输出完整需求清单
- 双方确认需求定稿

### 2.3 讨论中的关键原则

1. **不要一开始就讨论所有细节**——先定大方向，再逐层深入
2. **让 Claude 主动提问**——每轮结束让它说"你还有什么不清楚的需求？"，它会发现你没想到的问题
3. **每轮都做决策**——不要积累太多待定项，当场拍板
4. **功能分优先级**——P0/P1/P2 三级，MVP 只做 P0

### 2.4 产出：PRD 需求文档

最终的 `docs/PRD.md` 包含：

```
1. 产品概述（定位、技术栈、平台）
2. 核心功能（翻译触发、翻译能力、音标谐音）
3. 翻译源管理（离线词典、在线 API、AI 翻译源、缓存策略）
4. 弹窗交互（气泡样式、关闭方式、键盘导航）
5. 收藏功能（数据结构、列表、导出）
6. 历史记录
7. 发音朗读
8. 数据存储与同步（SQLite、坚果云、冲突处理、Migration）
9. 系统集成（托盘、外部 API、自动更新）
10. 设置项清单
11. 截图翻译（P2）
12. 首次启动引导
13. 优先级总览（P0/P1/P2）
14. 非功能性需求（性能、体积、内存、安全）
```

## 三、第二阶段：竞品对比与需求完善

### 3.1 操作方式

需求初稿完成后，**新开一个会话**，直接告诉 Claude：

> "查找网络上相关的项目进行对比现在的需求文档，找出不足之处"

Claude 会自动搜索 GitHub 上的同类开源项目，阅读它们的 README 和源码结构，然后和你的 PRD 逐项对比。

### 3.2 竞品对比矩阵

Claude 自动产出了一份详细的对比表：

| 维度 | **Pot Desktop** | **Bob** | **EchoWord（PRD 初稿）** |
|------|----------------|---------|------------------------|
| 技术栈 | Tauri + React | macOS 原生 Swift | Tauri + Web |
| 平台 | Win/Mac/Linux | macOS only | Win/Mac |
| 翻译引擎 | 20+（含 AI） | 30+（含 AI） | 3（腾讯/百度/有道） |
| 插件系统 | 完整 | 自定义插件 | 无 |
| OCR | 多引擎 | 多引擎 | P2 简单提及 |
| 生词本导出 | Anki/欧路/有道/扇贝 | 收藏夹 | CSV/JSON/Anki |

### 3.3 Claude 发现的问题（按严重程度分级）

#### 致命缺失（4 项）

| # | 问题 | Claude 的分析 |
|---|------|--------------|
| 1 | **缺少 AI/LLM 翻译源** | "2026 年的翻译工具，翻译源生态停留在 2022 年水平。" 竞品全部支持 DeepL、OpenAI、Gemini |
| 2 | **缺少多引擎对照翻译** | Bob 支持同时展示 10 个引擎结果，Pot 也支持多引擎并行。EchoWord 只能用一个 |
| 3 | **缺少剪贴板监听模式** | PDF 阅读、终端操作场景的刚需，Pot 有独立的「复制即翻译」模式 |
| 4 | **缺少外部调用能力** | Pot 暴露 HTTP API（`127.0.0.1:60828`），Bob 支持 PopClip。EchoWord 是信息孤岛 |

#### 重要缺失（6 项）

| # | 问题 | Claude 的分析 |
|---|------|--------------|
| 5 | **缺少开发者命名拆分** | Bob 支持 `getUserName` → `get User Name` 自动拆分翻译 |
| 6 | **缺少首次启动引导** | macOS 辅助功能权限申请的 UX 没设计，用户装完就懵 |
| 7 | **缺少翻译缓存** | 同一个单词查 10 次就请求 10 次 API |
| 8 | **缺少 API 限频/降级** | 百度 QPS=1 超限怎么办？快速连续划词的防抖策略没有 |
| 9 | **缺少网络代理配置** | 国内访问 DeepL/Google 需要代理 |
| 10 | **缺少自动更新** | Tauri 自带 updater，但 PRD 完全没提 |

#### 体验细节缺失（8 项）

弹窗位置记忆、键盘导航、隐私模式、UI 多语言、坚果云同步冲突处理、数据库 Migration、崩溃上报、Linux 平台支持。

#### EchoWord 的独有优势（竞品没有的）

Claude 同时指出了 PRD 中竞品**做不到**的差异化价值：

1. **音标中文谐音标注** — 全网独创，没有任何竞品有此功能
2. **分级离线词典** — 5 万/20 万/77 万三档按需下载，比 Pot 的一刀切更灵活
3. **坚果云同步方案** — 零后端成本，利用文件同步实现多设备

### 3.4 补充进 PRD 的功能

确认 Claude 的分析后，一句话让它执行：

> "把你上面提到的所有东西都写到 PRD 文档里吧，我都觉得挺不错的"

Claude 自动修改 PRD，新增/修改了 18 项内容：

| 补充功能 | 分析来源 | 优先级 |
|---------|---------|--------|
| DeepL 翻译源 | 致命缺失 #1 | P0 |
| OpenAI/Ollama AI 翻译 | 致命缺失 #1 | P1/P2 |
| 多引擎对照翻译 | 致命缺失 #2 | P1 |
| 剪贴板监听模式 | 致命缺失 #3 | P1 |
| 外部调用 HTTP API | 致命缺失 #4 | P1 |
| 开发者命名拆分 | 重要缺失 #5 | P1 |
| 首次启动引导 + 权限申请 | 重要缺失 #6 | P0 |
| 翻译缓存（LRU 500 条） | 重要缺失 #7 | P0 |
| 请求防抖 + API 降级链 | 重要缺失 #8 | P0 |
| 网络代理配置 | 重要缺失 #9 | P1 |
| 自动更新 | 重要缺失 #10 | P1 |
| 数据库 Migration | 体验细节 | P0 |
| 同步冲突处理 | 体验细节 | P1 |
| 弹窗键盘导航 | 体验细节 | P1 |
| 隐私模式 | 体验细节 | P2 |
| UI 多语言 i18n | 体验细节 | P2 |
| Linux 支持 | 体验细节 | P2 |
| 崩溃上报 | 体验细节 | P2 |

### 3.5 关键收获

竞品分析不是为了抄，而是为了：

1. **查漏补缺** — 从 PRD 初稿的 30 多项功能扩展到 50+ 项，补了 18 个盲点
2. **找到差异化** — 确认"音标中文谐音"确实是全网独创的市场空白
3. **学习架构** — 参考成熟项目的缓存、降级、插件化设计
4. **校准时代感** — Claude 一句"翻译源生态停留在 2022 年"直接点醒了 AI 翻译源的缺失

## 四、第三阶段：技术实现文档

### 4.1 操作方式

PRD 定稿后，**再开一个新会话**，让 Claude 进入 Plan Mode 编写技术文档：

> "根据现在的需求文档，制定一下技术实现文档吧"

关键指令：**只输出文档，不写代码**。这个阶段的目标是设计，不是实现。

### 4.2 技术选型确认

在生成技术文档之前，Claude 会先确认技术选型。这次的选型是：

### 4.4 开发周期规划

技术设计完成后，Claude 将所有功能拆分为 6 个开发周期，每个周期都是一个"可运行、可演示、可验收"的版本：

| 周期 | 名称 | 目标 | 关键交付 |
|------|------|------|---------|
| Cycle 01 | 基础工程与应用骨架 | Tauri 双窗口、数据库、设置、托盘骨架 | 可启动的桌面应用骨架 |
| Cycle 02 | 单词翻译闭环 | 离线词典、音标谐音、弹窗、收藏 | 第一个核心价值版本 |
| Cycle 03 | 句子翻译与触发链路 | 在线 API、缓存/降级、快捷键、输入翻译 | 主翻译链路可用 |
| Cycle 04 | P0 收口与发布候选 | 划词自动弹窗、权限引导、稳定性加固 | MVP Release Candidate |
| Cycle 05 | P1 体验增强 | 朗读、历史、主题、快捷键自定义 | 日常可持续使用版本 |
| Cycle 06 | 扩展集成与 P2 储备 | 词典管理、同步、OpenAI、HTTP API | 可扩展版本 |

每个周期都有独立的详细任务清单文档（`docs/cycles/cycle-0x-xxx.md`），包含具体的实现任务、验收标准和依赖关系。

## 五、第四阶段：开发周期规划

### 5.1 操作方式

技术设计完成后，在同一个会话中继续让 Claude 拆分开发周期：

> "读取需求文档和技术文档，分成多个开发周期，并给出对应周期的开发文档"

Claude 会基于 PRD 的优先级（P0/P1/P2）和技术设计的模块依赖关系，自动拆分出合理的开发周期。

### 5.2 拆分原则

Claude 遵循了以下拆分原则：

1. **每个周期都是可交付闭环** — 不是做完一个模块就算，而是每个周期结束后都有"可运行、可演示、可验收"的版本
2. **先打底座，再做业务** — Cycle 01 搭骨架，Cycle 02-04 做 P0 功能，Cycle 05-06 做 P1/P2
3. **依赖关系串行** — Cycle 01 → 02 → 03 → 04 → 05 → 06，每个周期依赖前一个的产出

### 5.3 产出物

```
docs/
├── development-roadmap.md         # 总览：6 个周期的目标、依赖关系、功能映射
└── cycles/
    ├── cycle-01-foundation.md     # 工程骨架：Tauri 双窗口、数据库、设置、托盘
    ├── cycle-02-word-translation.md  # 单词翻译：离线词典、音标谐音、弹窗、收藏
    ├── cycle-03-sentence-translation.md  # 句子翻译：在线 API、缓存/降级、快捷键
    ├── cycle-04-mvp-rc.md         # MVP 收口：权限引导、自动划词、稳定性加固
    ├── cycle-05-p1-experience.md  # 体验增强：朗读、历史、主题、快捷键自定义
    └── cycle-06-extensions.md     # 扩展集成：词典管理、同步、OpenAI、HTTP API
```

每个周期文档都包含：
- 具体的实现任务清单
- 验收标准
- 与其他周期的依赖关系
- 按 前端 / Rust 后端 / 联调 / 验收 分类的 TODO

## 七、第六阶段：进入项目，Claude 编排 + Codex 编码

### 7.1 多模型协作架构

从编码阶段开始，工作流切换为 **Claude + Codex 双模型协作模式**：

| 角色 | 模型 | 职责 | 工具 |
|------|------|------|------|
| **编排者** | Claude Code | 读取文档、规划任务、Review 代码、调度执行 | `/ccg:workflow`、`/ccg:feat`、`/ccg:review` |
| **执行者** | Codex CLI | 实际编码、文件创建/修改、代码分析 | `codeagent-wrapper.exe --backend codex` |

**协作原理：**
- Claude Code 通过 `ccg:workflow` 等技能调用 `codeagent-wrapper.exe`，将编码任务委派给 Codex
- Codex 在独立进程中执行，直接读写项目文件
- 支持会话复用（`resume SESSION_ID`），让 Codex 在多轮任务间保持完整上下文
- Claude 在 Codex 执行期间可以并行做其他工作（如更新文档、准备下一轮任务）

### 7.2 操作方式

进入项目目录启动 Claude Code：

```bash
cd /path/to/echo-word
claude
```

Claude Code 会自动读取 `CLAUDE.md`，理解项目背景。然后通过 `ccg:workflow` 或 `ccg:feat` 启动多模型协作流程：

> "读取 docs/PRD.md 和 docs/technical-design.md，按照 docs/cycles/cycle-01-foundation.md 开始开发"

Claude 会：
1. 读取需求文档和技术设计，理解要做什么、怎么做
2. 读取当前周期的任务清单，拆解为 Codex 可执行的任务
3. **调用 Codex** 实施编码，Codex 直接创建/修改项目文件
4. 等待 Codex 完成后，Claude 做代码 Review
5. 确认质量后 git commit

### 7.3 Codex 调用方式

Claude Code 通过后台任务调用 Codex：

```bash
# 新建会话
codeagent-wrapper.exe --backend codex - "$PWD" <<'EOF'
ROLE_FILE: prompts/codex/architect.md
<TASK>
需求：按 cycle-01 任务清单搭建工程骨架
上下文：Tauri v2 + React + TypeScript 项目
</TASK>
EOF

# 复用会话（保持上下文）
codeagent-wrapper.exe --backend codex resume <SESSION_ID> - "$PWD" <<'EOF'
<TASK>继续实现下一个任务</TASK>
EOF
```

关键参数：
- `--backend codex` — 指定使用 Codex 模型
- `resume SESSION_ID` — 复用之前的会话上下文
- `run_in_background: true` — 后台执行，Claude 可并行处理其他工作

## 八、第七阶段：持续迭代

### 8.1 一句"继续"的力量

每个周期开发完成后，只需要一句话推进下一个周期：

> "继续"

Claude 会自动读取下一个周期的任务清单，将编码任务委派给 Codex，衔接上一个周期的代码继续实现。

### 8.2 遇到问题时的处理

| 情况 | 处理方式 |
|------|---------|
| Codex 生成的代码有 bug | Claude Review 时发现，直接描述问题让 Codex 修复（会话复用） |
| 需求需要调整 | 先改 PRD，Claude 重新编排任务，再让 Codex 按新需求修改 |
| 技术方案需要变更 | 先改技术设计文档，Claude 重新规划，Codex 重构 |
| Codex 只输出 diff 不写文件 | 明确要求"直接修改文件，不要只输出 diff" |
| 上下文丢失 | 新开 Codex 会话，通过 CLAUDE.md 恢复上下文 |

### 8.3 每个周期结束后的检查

1. **Claude Review** — 通过 `/ccg:review` 让 Claude 审查 Codex 的代码
2. **验收标准** — 对照周期文档中的验收标准逐项确认
3. **git commit** — 确认质量后统一提交
4. **更新文档** — 更新 CLAUDE.md 记录当前进度

## 九、第九阶段：功能文档生成与评审优化循环

### 9.1 为什么需要这个阶段

编码实现阶段完成后，代码和原始文档之间必然存在偏差——有些功能简化了，有些扩展了，有些压根没做。如果不系统性地清算这些偏差，后续维护和迭代都会踩坑。

这个阶段的核心思路是：**让代码自己说话，再和文档对账**。

### 9.2 多模型协作模式

从这个阶段开始引入 Claude + Codex 双模型协作：

| 角色 | 模型 | 职责 |
|------|------|------|
| **规划者/评审者** | Claude Code | 对比文档差异、输出优化清单、评估完成度 |
| **执行者** | Codex（通过 codeagent-wrapper 调用） | 分析代码库、生成文档、实施代码修改 |

**协作工具链：**
- `codeagent-wrapper.exe` — Codex CLI 封装器，支持会话复用（`resume SESSION_ID`）
- `ccg:feat` / `ccg:workflow` — Claude Code 内置的多模型协作技能

### 9.3 操作流程

**步骤一：Codex 生成功能实现文档**

让 Codex 分析当前代码库，输出一份客观的功能实现文档（`docs/feature-implementation.md`），内容包括：
- 项目结构概览
- 已实现 / 部分实现 / 未实现的功能清单
- 核心功能流程描述
- 前后端交互说明
- 与原始 PRD 和技术设计的对比

> "让 Codex 读取项目结构和代码，生成完整的功能流程文档"

**步骤二：Claude 评审差异，输出优化清单**

Claude 对比 `feature-implementation.md` 和 `PRD.md`，按优先级给出优化清单：
- P0 致命缺失（MVP 不成立的问题）
- P1 体验缺口（影响可用性）
- 设置项"假生效"问题（存了但不生效，误导用户）
- 代码质量问题

**步骤三：Codex 按清单实施优化**

将优化清单发给 Codex，它直接修改代码文件。关键要求：
- 明确要求"直接修改文件，不要只输出 diff"
- 复用 Codex 会话（`resume SESSION_ID`），保持上下文连贯
- 每轮优化完成后 git commit

**步骤四：Codex 更新功能文档**

优化完成后，让 Codex 更新 `feature-implementation.md`，反映最新的代码状态。

**步骤五：Claude 再次评审**

重复步骤二，直到没有新的优化点（或剩余项都是 P2 低优先级）。

### 9.4 实际执行记录（EchoWord）

**第一轮优化（P0 致命缺失 + 设置假生效）：**

| 优化项 | 类型 | 结果 |
|--------|------|------|
| `translation-error` 事件闭环 | P0 | 后端翻译失败时 emit，popup 展示错误 |
| 弹窗坐标定位 | P0 | `show_popup_near_cursor` 跟随鼠标 + 边界检测 |
| Windows 选中自动弹窗 | P0 | 简化方案：检测 Ctrl+C → 读剪贴板 → 自动翻译 |
| `privacy_mode` 生效 | 假生效 | 开启时跳过 history 入库 |
| `auto_update` / `dictionary_version` / `data_dir` | 假生效 | UI 禁用并标注"即将支持" |

提交：`e350258 优化：补齐 P0 致命缺失与设置假生效纠偏`

**第二轮优化（P1 体验补齐）：**

| 优化项 | 类型 | 结果 |
|--------|------|------|
| 划词 300ms 防抖 | P1 | 快捷键回调加 debounce |
| 降级时 UI 提示 | P1 | 切换备用翻译源时附加 notice |
| 弹窗键盘导航 | P1 | Esc 关闭、Ctrl+C 复制、Tab 切换 |
| 收藏搜索 | P1 | 搜索框 + SQL LIKE 匹配 |
| 历史筛选 | P1 | 关键词 + 时间范围筛选 |

### 9.5 关键经验

1. **分轮次优化** — 先 P0 致命项，再 P1 体验项，不要一次塞太多
2. **Codex 会话复用** — `resume SESSION_ID` 让 Codex 保持完整上下文，不用每次重新分析
3. **每轮必须 git commit** — 方便回滚，也给下一轮评审提供清晰的 diff
4. **明确要求写入文件** — Codex 默认可能只输出 diff，必须显式要求"直接修改文件"
5. **评审收敛条件** — 剩余项全是 P2 或"未来版本"时可以停止循环

## 十、第十阶段：功能测试循环

### 10.1 操作流程

编码优化循环收敛后，进入测试阶段。同样采用 Claude + Codex 协作：

```
Claude 出测试清单 → Codex 执行排查 → 输出结果给 Claude → Claude 给新清单 → 循环直到通过
```

**测试维度：**

| 维度 | 关注点 |
|------|--------|
| 功能完整性 | 每个已实现功能是否按 PRD 预期工作 |
| 边界情况 | 空输入、超长文本、网络断开、API 限频 |
| 跨窗口交互 | 主窗口 ↔ popup 事件同步是否可靠 |
| 数据持久化 | 收藏/历史/设置的读写是否正确 |
| 性能 | 离线词典查询 < 50ms，缓存命中 < 5ms |
| 系统集成 | 快捷键、托盘、剪贴板监听在各平台的表现 |

### 10.2 测试清单模板

Claude 输出的测试清单格式：

```markdown
## 测试清单 - 第 N 轮

### 功能测试
- [ ] 快捷键翻译：选中文本 → Ctrl+C → 快捷键 → popup 弹出翻译结果
- [ ] 输入翻译：快捷键唤出输入框 → 输入文本 → 自动翻译
- [ ] 单词模式：输入 "ephemeral" → 显示释义 + 音标 + 谐音
...

### 异常测试
- [ ] 无网络时句子翻译 → 应显示错误提示
- [ ] API Key 为空时翻译 → 应触发降级链
...

### 回归测试
- [ ] 上一轮修复的 P0 项是否仍然正常
...
```

### 10.3 循环终止条件

- 所有 P0 功能测试通过
- P1 功能测试通过率 > 90%
- 无新发现的致命 bug
- 回归测试全部通过

## 十一、经验总结

### 11.1 什么时候该用 AI 对话

| 场景 | 适合 | 不适合 |
|------|------|--------|
| 需求头脑风暴 | ✅ AI 能快速发散，覆盖你没想到的点 | |
| 市场调研 | ✅ AI 能快速搜索和对比 | |
| 技术选型讨论 | ✅ AI 能对比利弊 | |
| 写文档 | ✅ AI 擅长结构化输出 | |
| 决策拍板 | | ❌ 最终决策必须是人做的 |
| 审美判断 | | ❌ UI/交互的"感觉"AI 把握不好 |

### 11.2 对话技巧

1. **先给场景，再问方案** — 不要上来就问"怎么做"，先描述你要解决什么问题
2. **让 AI 反问你** — "你还有什么不清楚的需求？"这句话能挖出很多隐藏需求
3. **逐轮收敛，不要发散** — 每轮对话结束时做决策，而不是堆积选项
4. **分优先级** — 不是所有功能都要 MVP 做完，P0/P1/P2 分清楚
5. **先需求后技术** — 需求没定稿之前不要急着写代码

### 11.3 文档驱动开发

整个流程的核心产出是**文档**，而不是代码：

```
PRD（做什么）→ 技术设计（怎么做）→ 开发周期（什么时候做）→ 代码实现
```

好处：
- 新人加入项目能快速理解上下文
- AI 能通过文档快速恢复上下文（CLAUDE.md）
- 需求变更时有据可查
- 避免"写到一半发现方向错了"的返工

### 11.4 多模型协作技巧

1. **明确角色分工** — Claude 做规划和评审，Codex 做实施和分析，不要让一个模型既规划又执行
2. **会话复用** — Codex 的 `resume SESSION_ID` 能省去大量重复的上下文加载
3. **每轮 commit** — 优化循环中每轮都 git commit，方便回滚和 diff 审查
4. **显式要求文件写入** — Codex 可能只输出 diff，必须明确说"直接修改文件"
5. **收敛条件要明确** — "没有 P0/P1 优化点"是停止循环的信号，不要无限追求完美
