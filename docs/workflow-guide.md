# 从零到一：用 Claude Code 驱动新产品的完整流程

> 本文以 **EchoWord**（轻量级划词翻译工具）为例，记录如何通过与 AI 对话，从一个模糊想法推进到可执行的技术方案。适合想了解"AI 辅助产品从 0 到 1"工作流的开发者。

---

## 一、整体流程概览

```
需求萌芽 → 市场调研 → 需求讨论 → 需求定稿 → 竞品对比 → 技术设计 → 开发规划 → 项目初始化
```

分为 4 个阶段：

| 阶段 | 做什么 | 产出物 |
|------|--------|--------|
| **第一阶段** | 和 Claude 来回对话，逐步明确需求 | `PRD.md`（需求文档） |
| **第二阶段** | 让 Claude 搜索竞品，对比找出差异化价值 | PRD 补充完善 |
| **第三阶段** | 让 Claude 输出技术实现文档 | `technical-design.md` |
| **第四阶段** | 创建项目，所有文档入库 | GitHub 仓库 + 完整文档 |

---

## 二、第一阶段：需求讨论（从模糊到清晰）

### 2.1 起点：一个朴素的需求

最初只是一个简单的问题：

> "有没有这样的软件——划词翻译，能显示音标，还能展示音标对应的中文谐音？因为我看不懂音标。"

这个阶段的关键是**不急着动手**，先和 Claude 充分讨论。

### 2.2 对话策略：逐轮收敛

需求讨论不是一次性完成的，而是通过多轮对话逐步收敛：

**第 1 轮：市场调研**
- 让 Claude 搜索现有翻译软件
- 结论：市面上没有"音标中文谐音"功能的翻译工具
- 决策：**这是一个值得做的差异化功能**

**第 2 轮：技术可行性**
- 问 Claude："如果我想做一个这样的小插件，可以吗？"
- Claude 给出三种技术方案（Electron / Tauri / 浏览器插件）及对比
- 决策：**选择 Tauri（轻量化）**

**第 3 轮：核心功能讨论**
- 数据同步方案（坚果云 WebDAV vs 本地文件同步）
- 收藏功能的数据结构
- 全局划词的触发方式
- 产品命名
- 决策：**坚果云本地同步、全局快捷键+鼠标选中双触发、命名 EchoWord**

**第 4 轮：细节确认**
- 翻译 API 选择（离线词典 + 在线 API 混合方案）
- 音标谐音的精度策略（混合标注方案）
- 弹窗交互形式（跟随鼠标气泡）
- 补充建议（发音朗读、历史记录、快捷键自定义、输入翻译）

**第 5 轮：最终确认**
- 逐项确认所有决策
- Claude 输出完整需求清单
- 双方确认需求定稿

### 2.3 讨论中的关键原则

1. **不要一开始就讨论所有细节**——先定大方向，再逐层深入
2. **让 Claude 主动提问**——每轮结束让它说"你还有什么不清楚的需求？"，它会发现你没想到的问题
3. **每轮都做决策**——不要积累太多待定项，当场拍板
4. **功能分优先级**——P0/P1/P2 三级，MVP 只做 P0

### 2.4 产出：PRD 需求文档

最终的 `docs/PRD.md` 包含：

```
1. 产品概述（定位、技术栈、平台）
2. 核心功能（翻译触发、翻译能力、音标谐音）
3. 翻译源管理（离线词典、在线 API、AI 翻译源、缓存策略）
4. 弹窗交互（气泡样式、关闭方式、键盘导航）
5. 收藏功能（数据结构、列表、导出）
6. 历史记录
7. 发音朗读
8. 数据存储与同步（SQLite、坚果云、冲突处理、Migration）
9. 系统集成（托盘、外部 API、自动更新）
10. 设置项清单
11. 截图翻译（P2）
12. 首次启动引导
13. 优先级总览（P0/P1/P2）
14. 非功能性需求（性能、体积、内存、安全）
```

## 三、第二阶段：竞品对比与需求完善

### 3.1 让 Claude 搜索同类项目

需求初稿完成后，让 Claude 搜索 GitHub 上的同类项目进行对比：

- **Pot 翻译** — 跨平台划词翻译，功能丰富
- **Bob** — macOS 翻译工具
- **CopyTranslator** — 复制即翻译
- 其他开源翻译工具

### 3.2 对比后补充的功能

通过竞品分析，发现了原始需求中遗漏的点，补充进 PRD：

| 补充功能 | 来源 | 优先级 |
|---------|------|--------|
| 剪贴板监听模式 | 竞品 CopyTranslator 的核心功能 | P1 |
| 开发者命名拆分（camelCase/snake_case） | 开发者场景特有需求 | P1 |
| 多引擎对照翻译 | Pot 翻译的特色功能 | P1 |
| DeepL 翻译源 | 翻译质量最佳的免费 API | P0 |
| AI 翻译源（OpenAI/Ollama） | 新一代翻译趋势 | P1/P2 |
| 翻译缓存 + 请求防抖 + API 降级链 | 工程健壮性 | P0 |
| 外部调用 HTTP API | 与 Alfred/Raycast 等工具联动 | P1 |
| 自动更新 | 利用 Tauri 内置 updater | P1 |
| 首次启动引导 + 权限申请 | 用户体验 | P0 |
| 数据库 Migration 机制 | 数据稳定性 | P0 |
| 同步冲突处理 | 坚果云多设备场景 | P1 |

### 3.3 关键收获

竞品分析不是为了抄，而是为了：
1. **查漏补缺** — 发现自己没想到的功能点
2. **找到差异化** — 确认"音标中文谐音"确实是市场空白
3. **学习架构** — 参考成熟项目的缓存、降级、插件化设计

## 四、第三阶段：技术实现文档

### 4.1 让 Claude 输出技术设计

需求定稿后，让 Claude 基于 PRD 生成 `docs/technical-design.md`，内容包括：

```
1. 技术栈与依赖清单（Rust 后端 + React 前端，精确到版本号）
2. 项目结构（目录规划）
3. Tauri 双窗口架构设计
4. 数据库 Schema（表结构 + Migration 策略）
5. 翻译源抽象层（Trait 接口设计）
6. 音标谐音映射表
7. IPC 通信协议（前端 ↔ Rust 后端）
8. 缓存与限频策略
9. 系统集成细节（快捷键、托盘、权限）
10. 安全设计（API Key 加密）
```

### 4.2 开发周期规划

技术设计完成后，让 Claude 将所有功能拆分为 6 个开发周期：

| 周期 | 名称 | 目标 |
|------|------|------|
| Cycle 01 | 基础工程与应用骨架 | Tauri 双窗口、数据库、设置、托盘骨架 |
| Cycle 02 | 单词翻译闭环 | 离线词典、音标谐音、弹窗、收藏 |
| Cycle 03 | 句子翻译与触发链路 | 在线 API、缓存/降级、快捷键、输入翻译 |
| Cycle 04 | P0 收口与发布候选 | 划词自动弹窗、权限引导、稳定性加固 |
| Cycle 05 | P1 体验增强 | 朗读、历史、主题、快捷键自定义 |
| Cycle 06 | 扩展集成与 P2 储备 | 词典管理、同步、OpenAI、HTTP API |

每个周期都有独立的任务清单文档（`docs/cycles/cycle-0x-xxx.md`）。

## 五、第四阶段：项目初始化

### 5.1 创建 GitHub 仓库

```bash
# 安装 GitHub CLI
brew install gh

# 登录
gh auth login

# 创建仓库并克隆
gh repo create echo-word --public --description "..." --clone --license mit

# 移动到项目目录
mv echo-word /path/to/project/
```

### 5.2 项目文档结构

```
echo-word/
├── CLAUDE.md                          # AI 上下文文件（项目概述 + 关键决策）
├── LICENSE
├── .gitignore
└── docs/
    ├── PRD.md                         # 产品需求文档
    ├── technical-design.md            # 技术实现文档
    ├── development-roadmap.md         # 开发周期规划总览
    └── cycles/                        # 各周期详细任务清单
        ├── cycle-01-foundation.md
        ├── cycle-02-word-translation.md
        ├── cycle-03-sentence-translation.md
        ├── cycle-04-mvp-rc.md
        ├── cycle-05-p1-experience.md
        └── cycle-06-extensions.md
```

### 5.3 CLAUDE.md 的作用

项目根目录的 `CLAUDE.md` 是给 Claude Code 读的上下文文件。它记录了：
- 技术栈选型
- 关键决策记录
- 当前阶段状态
- 文档索引

这样在任何新会话中打开这个项目，Claude 都能立即理解项目背景，无需重复解释。

## 六、经验总结

### 6.1 什么时候该用 AI 对话

| 场景 | 适合 | 不适合 |
|------|------|--------|
| 需求头脑风暴 | ✅ AI 能快速发散，覆盖你没想到的点 | |
| 市场调研 | ✅ AI 能快速搜索和对比 | |
| 技术选型讨论 | ✅ AI 能对比利弊 | |
| 写文档 | ✅ AI 擅长结构化输出 | |
| 决策拍板 | | ❌ 最终决策必须是人做的 |
| 审美判断 | | ❌ UI/交互的"感觉"AI 把握不好 |

### 6.2 对话技巧

1. **先给场景，再问方案** — 不要上来就问"怎么做"，先描述你要解决什么问题
2. **让 AI 反问你** — "你还有什么不清楚的需求？"这句话能挖出很多隐藏需求
3. **逐轮收敛，不要发散** — 每轮对话结束时做决策，而不是堆积选项
4. **分优先级** — 不是所有功能都要 MVP 做完，P0/P1/P2 分清楚
5. **先需求后技术** — 需求没定稿之前不要急着写代码

### 6.3 文档驱动开发

整个流程的核心产出是**文档**，而不是代码：

```
PRD（做什么）→ 技术设计（怎么做）→ 开发周期（什么时候做）→ 代码实现
```

好处：
- 新人加入项目能快速理解上下文
- AI 能通过文档快速恢复上下文（CLAUDE.md）
- 需求变更时有据可查
- 避免"写到一半发现方向错了"的返工
